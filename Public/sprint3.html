Sprint 3

Introdução

Consequência de uma conferência internacional no Dubai, foi-nos proposto efetuar algumas mudanças na CASE Tool a implementar. Sumariamente deixarão de existir Fases de Projeto, passando as Tarefas a estar diretamente ligadas ao mesmo. Sendo assim, os casos de utilização anteriormente associados a Fases, passam a corresponder às Tarefas. Também podemos agora agrupar as Tarefas por Templates, que correspondem a blocos de tarefas que pertencem a funcionalidades idênticas, ou que estão interligadas num determinado projeto.
Além destas alterações, será introduzida a Gestão de Qualidade dos Projetos, que tem como função avaliar e corrigir Defects que possam surgir.
A nosso ver, a mudança mais importante refere-se à Gestão Temporal da Informação, onde nos é proposto analisar e concluir se é benéfico implementar Bases de dados/Tabelas temporais na nossa CASE Tool. A análise passa por responder a questões como:

  • Quando devemos implementar?
  • Como fazê-lo?
  • Construir os mecanismos sobre as bases de dados existentes ou utilizar um dos poucos motores de bases de dados que já suportam as tabelas temporais?
  • As soluções implementadas são boas?
  • Queremos ficar dependentes de uma tecnologia?
  • É demasiado esforço “construir a roda” implementando de raiz a gestão temporal?

Todas estas questões e problemas serão respondidas ao longo deste Sprint.
Posto isto, o foco desta entrega é modelar novos UML, tendo em conta os novos requisitos, e avaliar a viabilidade do uso de Bases de Dados Temporais na nossa CASE Tool. Os diagramas essenciais a apresentar são:

  • Diagrama de classes;
  • Casos de utilização (use cases);
  • Diagrama de máquina de estados;

Gestão Temporal da Informação

Na introdução deste Sprint referimos que a nosso ver, a mudança mais importante refere-se à Gestão Temporal da Informação. Tal mudança passaria por implementar uma Bases de dados/Tabelas temporais na nossa CASE Tool.
A nosso ver faria sentido separar da restante documentação, a análise desta possibilidade, tendo em conta que é a mudança mais impactante no software a criar. Posto isto, é importante compreender o que são bases de dados temporais, e ter uma breve noção de como funcionam.

O que são Bases de Dados Temporais:

Uma base de dados temporal armazena dados relacionados a instâncias de tempo. Oferece tipos de dados temporais e armazena informações relacionadas ao passado, presente e futuro. As bases de dados temporais podem ser uni-temporais, bi-temporais ou tri-temporais.
Mais especificamente, os aspectos temporais geralmente incluem tempo válido, tempo de transação ou tempo de decisão:

  • Tempo válido é o período durante o qual um fato é verdadeiro no mundo real;
  • Tempo de transação é o período durante o qual um fato armazenado no banco de dados era conhecido;
  • Tempo de decisão é o período durante o qual um fato armazenado no banco de dados foi decidido como válido;


    • A nosso ver, uma base de dados bi-temporal é a mais adequada para a nossa CASE Tool. A sua implementação não é tão complexa como a tri-temporal, e é mais completa e mais eficiente relativamente à uni-temporal. Além disso tendo em conta a complexidade de implementação de uma base de dados tri-temporal, e ainda as necessidades do nosso produto, achamos que uma base de dados bi-temporal é suficiente.
    Este tipo de bases de dados irá permitir termos acesso a todo o historial de dados alguma vez introduzidos/alterados, pois quando é feito um update, os novos dados são introduzidos, e os anteriores guardados numa tabela temporal. Sendo assim, é uma forma de obter dados do passado, pervalencendo assim toda a informação que a nossa CASE Tool deteve. Isto permite podermos recuperar estados prévios de tabelas, caso necessário, e permite uma melhor previsão do futuro e visualização de todos os processos que ocorreram.

    Base de Dados bi-temporal:

    Nas bases de dados bi-temporal além do momento em que as operações CRUD (Create, Read, Update, Delete) são registadas na base de dados (“transaction time”), também associamos uma outra dimensão chamada “valid time” (ou “belief time”) que significa a que espaço temporal é que os dados que estamos a indicar correspondem.

    Nossa abordagem relativamente a este tema:

    Uma das questões colocadas relativamente a esta alteração, passaria por analisar qual a melhor data de implementação da mesma. Isto é, se faria mais sentido implementar esta Gestão Temporal agora na fase inicial do Projeto, ou se nos últimos Sprints quando todo o software já estivesse criado.
    A melhor opção destas duas, é logicamente, implementar a base de dados temporal nesta fase inicial em que nos encontramos. Migrar uma base de dados já é um processo complexo e demorado, se adicionarmos o factor temporal, torna-se ainda mais complicado. Sendo assim, e tendo em conta que concordamos e achamos que esta mudança é uma mais valia para o nosso projeto, a criação da base de dados temporal será feita nos próximos Srints, tendo em conta que a original ainda não estava totalmente finalizada. Desta forma conseguimos aqdequar o nosso Software à base de dados em questão desde início, poupando imenso tempo e esforço com migrações de uma base de dados não temporal para uma deste tipo.

    Relativamente à sua implementação, embora seja uma tecnologia bastante recente, alguns motores de bases de dados já implementaram temporal tables. Estivemos a pesquisar e encontramos uma excelente opção gratuita, o Sistema Gestor de Bases de Dados MariaDB. MariaDB é um avançado substituto para o MySQL e está disponível sob os termos da licença gratuita GPL v2.
    Ao optarmos por escolher um motor de base de dados com esta funcionalidade já implementada, evitamos “construir a roda”, pois o nível de esforço e complexidade de criar a gestão temporal de raiz, ou seja, implementar de raiz as tabelas temporais é mais elevada. Além disso, construir de raiz está mais propício a erros relacionais, mais uma razão pela qual devemos utilizar o SGBD MariaDB para alcançar a Gestão Temporal desejada.

    Uma das objeções à implementação desta tecnologia é a dependência da mesma. Uma base de dados não temporal também origina certas dependências, nomeadamente a dependência do SGBD inicialmente escolhido, isto é, caso fosse necessário alterar no futuro o motor de base de dados escolhido, iria ser necessário investir algum esforço para migrar do SGBD original para o novo. Da mesma forma, uma base de dados temporal apresenta exatamente a mesma dependência, e tendo em conta que é uma tecnologia em crescimento, no futuro surgirão mais opções de escolha de SGBD's com esta capacidade e o tempo e esforço de migração será provavelmente idêntico ao de uma base de dados não temporal. Além disso os benefícios relativos à Gestão Temporal, e superioridade tecnológica e funcional destas bases de dados temporais relativamente à que originalmente ponderávamos implementar, compensa o risco de dependência.

    Conclusões

    Consequência da elaboração de todos estes Diagramas e inclusive análise dos requisitos para a criação da CASE Tool, a nossa equipa considera a sua implementação viável. O foco estaria em criar uma ferramenta eficiente e que permitisse criar uma gestão total do projeto a partir da mesma, do mais baixo ao mais alto nível. Posto isto, aguardamos por novas instruções para dar continuidade ao nosso trabalho.

    Após várias reuniões chegamos ainda à conclusão que a melhor forma de criar este projeto seria utilizando HTML/CSS e Javascript, não havendo assim necessidade de realizar o download do software para cada computador da empresa e permitindo realizar todo o trabalho no Browser. O mesmo será acedido através da nosso Website.
